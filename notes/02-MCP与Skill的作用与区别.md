# 📖 概念笔记：MCP vs Skill

> **一句话**：MCP = USB-C通用接口标准（谁都能用）；Skill = 专用外设驱动（特定功能）
> **Day 10必须实际安装一个MCP Server，哪怕只是天气查询。**

---

## 🔌 MCP（Model Context Protocol）是什么？

**定义**：让大模型/Agent和外部工具"对话"的**通用协议标准**。

```
就像 USB-C 接口：
- 任何品牌的充电器（模型）→ USB-C（MCP协议）→ 任何品牌的手机（工具）
- 不需要为每个组合单独做一根线
```

### MCP的架构

```
┌──────────┐     MCP协议      ┌──────────────┐
│ AI Agent │ ←─────────────→ │  MCP Server   │
│ (客户端)  │   标准化通信     │ (工具提供方)  │
└──────────┘                  └──────────────┘
                                   │
                              ┌────┴────┐
                              │ 天气查询  │
                              │ 文件读写  │
                              │ 数据库查询│
                              └─────────┘
```

---

## 📋 Skill 是什么？

**定义**：为特定Agent平台编写的**专用功能包**，包含完整的操作流程。

```
就像打印机的专用驱动：
- 为某个Agent定制的操作指南
- 告诉Agent：遇到X情况 → 用Y方法 → 输出Z结果
- 比MCP更"重"：不仅定义接口，还定义全流程
```

### Skill vs MCP 核心对比

| 对比维度 | MCP | Skill |
|:---|:---|:---|
| **本质** | 通信协议标准 | 功能模块包 |
| **类比** | USB-C 接口 | 打印机驱动 |
| **安装方式** | 配置连接参数 | 安装功能包 |
| **调用协议** | 标准化JSON-RPC | 平台特定（各Agent不同） |
| **复用范围** | **跨平台**—任何支持MCP的Agent都能用 | **单平台**—只为特定Agent设计 |
| **开发者视角** | 写一次，到处用 | 每个平台写一份 |
| **粒度** | 细粒度—一个工具 | 粗粒度—一个完整工作流 |
| **例子** | "查天气"这一个工具 | "数据分析全流程：读数据→清洗→画图→写报告" |

---

## 🔄 它们如何协同工作？

```
用户：帮我查下这个报告里的销售额趋势

Agent（大模型）：
  → 识别意图："需要查报告"+"分析趋势"
  → 选工具：MCP里的"文件读取工具"读出报告
  → 执行Skill："数据分析全流程"
      ├── 步骤1：解析数据（Skill定义的方法）
      ├── 步骤2：计算趋势（Skill定义的方法）
      └── 步骤3：生成可视化（调用MCP里的"画图工具"）

结论：MCP提供"原子工具"，Skill编排"工作流程"
```

## 🧪 测试角度

### 测MCP兼容性

| 测试用例 | 方法 |
|:---|:---|
| 同一MCP Server给不同Agent用 | Agent A 和 Agent B 调同一个MCP天气工具，看结果是否一致 |
| 协议边界 | 传一个超长城市名，看MCP Server是否优雅处理 |
| 断连恢复 | 调MCP时断网，看Agent是否能发现并报错 |

### 测Skill边界

| 测试用例 | 方法 |
|:---|:---|
| 必填参数缺失 | 不传城市名直接调天气Skill，看是否给出明确提示 |
| 参数类型错误 | 城市名传数字"123"，看是否识别并提示 |
| 工具不可用 | Skill依赖的MCP工具挂了，看Skill是否优雅失败 |
| 结果验证 | Skill推荐的穿衣建议是否和天气数据吻合 |
